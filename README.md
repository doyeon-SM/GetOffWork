# 🏢 퇴근하고싶다 (GetOffWork)

> *"오늘도 제발 제시간에 퇴근하게 해주세요..."*

민원 서비스 데스크에서 공무원이 되어 밀려드는 민원인들을 상대하는 **데스크 시뮬레이션 게임**

<br>

---

## 🎮 게임 소개

당신은 오늘부터 민원실 직원입니다.  
메뉴얼에 따라 민원인을 정확하게 응대하고, 스트레스를 버티며, 무사히 퇴근하세요.

잘못된 절차, 진상 민원인, 쌓이는 스트레스... 퇴근은 쉽지 않습니다.

<br>

---

## 🕹 핵심 게임플레이

### 📋 메뉴얼 기반 응대 시스템

민원 종류마다 정해진 **절차 메뉴얼**이 있습니다.  
플레이어가 실행한 행동은 **Queue**에 기록되고, 메뉴얼 **List**와 비교해 정확도를 판정합니다.

- **절차 누락** → 패널티
- **절차 위반** → 패널티
- **불필요한 절차** → 패널티
- **정상 응대 완료** → 성과 상승

현재 구현된 민원 유형:
| 민원 유형 | 설명 |
|---|---|
| `FullID_Self` | 본인 신분증 확인 및 서류 발급 |
| `FullID_Proxy` | 대리인 신분증 확인 및 서류 발급 |
| `AddressChange` | 본인 신분증 주소 변경 |
| `NewID` | 새로운 주민 증록 |

<br>

### 😤 진상 민원인 시스템

민원인의 **10%** 확률로 `Rudely` 타입과 **10%** 확률로 `Talkative` 타입의 진상 민원인이 등장합니다.

- 모든 대사가 **막말**로 바뀜
- 대화가 출력될 때마다 **스트레스 +1**

- 모든 대사가 **길어지고**
- 대화가 출력될 때마다 **스트레스 +1**

- 유형별 확률 및 패널티는 ScriptableObject로 에디터 조절 가능

<br>

### 🆘 진상 대응 메뉴얼 (SOS)

감당이 안 될 때는 **SOS를 발동**해 강제 종료할 수 있습니다.

| 스탯 | 변화 |
|---|---|
| 성과 | -2 |
| **스트레스** | -5 |
| 친절도 | -5 |

<br>

### 📊 플레이어 스탯

| 스탯 | 설명 |
|---|---|
| **Performance** | 응대 성공/실패에 따라 변화하는 성과 수치 |
| **Kindness** | 응대 태도와 진상 노출 횟수에 영향을 받음 |
| **Reliability** | 절차 누락/불필요 절차 시 감소 |
| **Stress** | 진상 민원인 응대, 실수 시 증가 |
| **Pay** | 편의점에서 아이템 구매에 사용하는 급여 |

하루가 끝나면 **결과지**에 스탯 변화량이 출처별로 시각화되어 표시됩니다.

<br>

### 🏪 편의점 & 인벤토리

점심시간에 편의점에서 아이템을 구매해 스탯을 관리할 수 있습니다.

- 매 방문마다 **4개의 아이템이 무작위 진열**
- 인벤토리 슬롯 **최대 3개**
- 같은 아이템은 하루에 **1개만 구매** 가능
- `2_MainScene`에서만 사용 가능

<br>

---

## 🗓 하루 구조

```
출근(편의점 방문 선택) → 오전 근무 → 점심시간 → 오후 근무 → 퇴근 결과지
```

- **오전/오후** 두 페이즈로 나뉜 근무
- 영업 종료 시 대기 중인 민원인은 자동 취소 (스탯 패널티 발생)
- 하루 결산 후 **DayResultView**에서 스탯 변화 확인

<br>

---

## 🏗 시스템 구조

```
민원인 대기 큐 → 호출 → 메뉴얼 실행 → 결과 판정 → 스탯 반영 → 결과 기록
```

### 주요 시스템 구성

| 시스템 | 설명 |
|---|---|
| `ServiceDeskManager` | 민원인 호출, 응대 흐름 전체 관리 |
| `ServiceDataManager` | 주민 데이터, 대사 테이블, 메뉴얼 SO 싱글톤 관리 |
| `ManualDataSO` | 에디터에서 편집 가능한 메뉴얼 정의 |
| `ManualEvaluator` | Queue/List 비교를 통한 절차 판정 |
| `ComplaintContext` | 민원인 유형·성격(NuisanceType) 정보 보유 |
| `DayResultData` | 하루 스탯 변화를 `StatChangeEvent` 큐로 기록 |
| `PlayerBase` | 스탯 저장 및 `AddStat()` 인터페이스 제공 |
| `WorkDayManager` | 하루 Phase 관리 (오전 / 점심 / 오후) |

<br>

### 대사 체계

```
입장 대사  : ComplaintOpeningLineTable  (민원 유형 × 신청인 유형 × 진상 유형 조합)
응대 대사  : CommandDialogueSO          (ManualDataSO에서 commandId로 조회)
퇴장 대사  : ComplaintClosingLineTable  (입장과 동일한 조합 구조)
```

<br>

## 🧩 오브젝트 시스템

드래그 & 드롭 기반 실물 서류 처리 시스템입니다.

| 오브젝트 | 설명 |
|---|---|
| `DeskObjectItem` | 드래그/드롭 가능한 아이템 베이스 클래스 |
| `IDCardItem` | 신분증 아이템. TakeZone에 드롭해 반납 |
| `PaperItem` | 인쇄된 서류. 열람 후 반납 필요 |
| `ObjectManagerBox` | 아이템 생성/반납 검사/영역 관리 |
| `TakeObjectZone` | 신분증·서류 제출 및 반납 서브 영역 |

<br>

## 📖 튜토리얼

첫째 날에는 튜토리얼이 자동 시작됩니다.

```
1일차 감지 → 예/아니오 팝업
  → 수락 시 강제 손님 삽입
       → Intro Flow → Print Flow → Mobile Flow → 종료
```

- `TutorialFlowSO`로 메뉴얼별 튜토리얼 단계를 분리 정의
- 특정 CommandId 실행 감지로 다음 단계 자동 진행
- `TutorialHighlighter`로 UI 강조 표시
